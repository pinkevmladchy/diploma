# Контекст проєкту «Smart Home» для написання дипломної роботи

> Цей документ — повний технічний опис інформаційної системи управління розумним
> домом. Призначений для передачі в LLM (claude.ai) як вихідний матеріал для
> генерації тексту пояснювальної записки дипломної роботи. Усі факти зібрані
> безпосередньо з вихідного коду — нічого не вигадано.

---

## 1. Загальний опис і мета системи

**Smart Home** — це веб-орієнтована інформаційна система для управління «розумним
домом». Система реалізує повну ієрархію об'єктів `Користувач → Будинок → Кімната →
Пристрій`, збирає телеметрію з IoT-пристроїв, у реальному часі оновлює інтерфейс
через WebSocket, надає аналітику з кількома типами графіків, підтримує правила
алертів (порогові сповіщення) з історією подій та сценарії автоматизації.

**Призначення:** централізований моніторинг і керування пристроями домашньої
автоматизації (термостати, лампи, датчики руху, лічильники електроенергії,
датчики якості повітря, датчики протікання, розумні замки), із візуалізацією
показників та автоматичним реагуванням на події.

**Ключова особливість — дворівнева рольова модель:**
- **admin** (адміністратор) — автоматично створюється при першому запуску;
  керує обліковими записами клієнтів, переглядає статистику, може «увійти як»
  клієнт (impersonation);
- **user / customer** (клієнт) — реєструється самостійно; володіє повним доменом
  розумного дому (будинки, кімнати, пристрої, телеметрія, аналітика, сценарії,
  алерти). Адміністратор навмисно **не має доступу** до доменних ресурсів.

### 3.2. Потоки даних

**Потік телеметрії (real-time):**
1. Емулятор (таймер per-user, інтервал 1–60 с) генерує нове значення для кожного
   пристрою користувача.
2. Пакетна вставка записів у таблицю `telemetry`.
3. Виклик `alertEvaluator` — перевірка порогів, відкриття/закриття подій алертів.
4. Виклик `evaluateSensorScenarios` — спрацювання сенсорних сценаріїв (rising-edge).
5. Емісія WebSocket-події `telemetry:new` у кімнату `user:<id>`.
6. Фронтенд (графіки пристрою, лог телеметрії) додає нові точки без перезапиту.

**Потік алертів:**
- Після кожного запису телеметрії `alertEvaluator` порівнює значення з активними
  правилами; при першому перетині порога (rising-edge) створює `AlertEvent` і
  емітить `alert:event` (kind `opened`); коли умова більше не виконується — закриває
  подію (`clearReason='auto'`) і емітить `cleared`. Користувач може закрити вручну
  (`clearReason='manual'`).

## 4. Модель даних (база даних)

PostgreSQL 16, ORM — Prisma. **11 таблиць.** Каскадне видалення по ланцюгу
`User → House → Room → Device → Telemetry`.

> Примітка: коментар у `schema.prisma` помилково згадує «9 таблиць» — це застарілий
> коментар; фактично таблиць 11 (пізніше додані `alert_events` і `notifications`).

### 4.1. Перелік таблиць

| # | Таблиця | Призначення | Ключові поля |
|---|---|---|---|
| 1 | **users** | користувачі та ролі | `id` (UUID), `email` (unique), `password_hash`, `full_name`, `role` (admin/user), `avatar_url`, `theme_brand`, `theme_primary`, `created_at` |
| 2 | **houses** | будинки, належать користувачу | `id` (serial), `user_id` (FK→users, cascade), `name`, `address` |
| 3 | **rooms** | кімнати в будинку | `id`, `house_id` (FK→houses, cascade), `name`, `description`, `floorplan_url` |
| 4 | **devices** | пристрої в кімнаті | `id` (UUID), `room_id` (FK→rooms, cascade), `name`, `type`, `status` (on/off), `is_online`, `floorplan_x`, `floorplan_y` (нормалізовані координати [0..1] на плані) |
| 5 | **telemetry** | сирі вимірювання | `id` (BigInt), `device_id` (FK→devices, cascade), `metric_type`, `value` (Decimal 14,4), `unit`, `timestamp` |
| 6 | **telemetry_aggregated** | hourly/daily агрегати | `device_id`, `metric_type`, `period` (hour/day), `avg/min/max_value`, `period_start`; unique-обмеження на бакет |
| 7 | **device_logs** | історія керування пристроями | `device_id`, `user_id` (FK→users, set null), `action`, `old_value`, `new_value`, `timestamp` |
| 8 | **scenarios** | сценарії автоматизації | `user_id` (cascade), `name`, `trigger_type` (time/sensor/manual), `trigger_value` (JSONB), `actions` (JSONB), `is_active` |
| 9 | **alerts** | правила порогів (рівень будинку) | `house_id` (cascade), `name`, `metric_type`, `condition` (gt/lt/eq/gte/lte), `threshold_value`, `is_active` |
| 10 | **alert_events** | події перетину порога (відкриті/закриті) | `alert_id`, `device_id`, `metric_type`, `trigger_value` (значення першого перетину), `latest_value`, `unit`, `threshold_value`, `condition`, `triggered_at`, `last_seen_at`, `cleared_at`, `clear_reason` (auto/manual) |
| 11 | **notifications** | сповіщення (від дій сценаріїв `notify`) | `user_id` (cascade), `device_id` (set null), `message`, `type` (warning/info/alert), `is_read`, `created_at` |

### 4.2. Перелічення (enum)

- **UserRole**: `admin`, `user`
- **DeviceType**: `thermostat`, `lamp`, `motion_sensor`, `power_meter`,
  `air_quality`, `water_leak`, `smart_lock`
- **DeviceStatus**: `on`, `off`
- **MetricType**: `temperature`, `humidity`, `power`, `motion`, `co2`,
  `light_level`, `water_leak`
- **AggregationPeriod**: `hour`, `day`
- **ScenarioTriggerType**: `time`, `sensor`, `manual`
- **AlertCondition**: `gt` (>), `lt` (<), `eq` (==), `gte` (>=), `lte` (<=)
- **NotificationType**: `warning`, `info`, `alert`

### 4.3. Індекси (продуктивність)

- `telemetry`: `(device_id, timestamp DESC)` — гарячий шлях для запитів останніх
  значень і діапазонів; `(metric_type, timestamp DESC)`.
- `telemetry_aggregated`: unique `(device_id, metric_type, period, period_start)`.
- `alert_events`: `(alert_id, cleared_at, triggered_at DESC)`, `(device_id, cleared_at)`.
- `notifications`: `(user_id, created_at DESC)`.
- FK-індекси на `houses.user_id`, `rooms.house_id`, `devices.room_id`,
  `alerts.(house_id, metric_type)`, `scenarios.user_id`.

### 4.4. Зв'язки та цілісність

- Каскадне видалення вниз по ланцюгу володіння: видалення користувача прибирає
  всі його будинки → кімнати → пристрої → телеметрію.
- `device_logs.user_id` та `notifications.device_id` — `ON DELETE SET NULL`
  (зберігається історія навіть після видалення пов'язаного об'єкта).

### 4.5. Історія міграцій (6 міграцій)

1. `init` — початкова схема.
2. `add_floorplan` — поля плану кімнати та координати пристроїв.
3. `remove_camera_type` — видалення типу пристрою «камера».
4. `add_alert_events` — таблиця подій алертів (історія перетинів порогів).
5. `add_user_avatar` — аватар користувача.
6. `add_user_theme` — збереження теми (brand/primary) per-user.

## 5. Backend: структура та REST API

### 5.1. Структура

```
backend/src/
├── index.ts              # bootstrap: маршрути, graceful shutdown
├── db.ts                 # Prisma client singleton
├── realtime.ts           # Socket.io (JWT-handshake, кімнати user:<id>)
├── uploads.ts            # multer: avatars/ (до 2 МБ), floorplans/ (до 5 МБ)
├── auth/
│   ├── jwt.ts            # sign/verify access & refresh токенів
│   └── middleware.ts     # requireAuth, requireRole
├── bootstrap/
│   ├── seedAdmin.ts      # ensureAdminUser (ідемпотентно)
│   └── seedDemoCustomer.ts # ensureDemoCustomer + демо-дані
├── services/
│   ├── emulator.ts       # генератор телеметрії per-user
│   ├── alertEvaluator.ts # оцінка порогів, події алертів
│   └── scenarioEngine.ts # час + сенсорні тригери
└── routes/               # auth, houses, rooms, devices, telemetry,
                          # alerts, scenarios, emulator, admin
```

Маршрути монтуються в корені (`/auth`, `/houses`, …), **без** префікса `/api`.
Vite-proxy на фронтенді переписує `/api/*` → `:4000` (відрізаючи `/api`).

### 5.2. Повний перелік ендпоінтів REST API

**Auth (`/auth`):**
- `POST /auth/register` — реєстрація (тільки role=user)
- `POST /auth/login` — вхід (повертає access+refresh+user)
- `POST /auth/refresh` — оновлення access-токена
- `GET /auth/me` — поточний користувач
- `PATCH /auth/me/theme` — збереження теми (brand/primary)
- `POST /auth/me/avatar` — завантаження аватара (multer)
- `DELETE /auth/me/avatar` — видалення аватара

**Houses (`/houses`, role=user):**
- `GET /houses`, `GET /houses/:id`, `POST /houses`, `PATCH /houses/:id`, `DELETE /houses/:id`

**Rooms (`/rooms`, role=user):**
- `GET /rooms`, `GET /rooms/:id`, `POST /rooms`, `PATCH /rooms/:id`, `DELETE /rooms/:id`
- `POST /rooms/:id/floorplan` (завантаження плану, multer), `DELETE /rooms/:id/floorplan`

**Devices (`/devices`, role=user):**
- `GET /devices`, `GET /devices/:id`, `GET /devices/:id/telemetry` (серія для графіка)
- `POST /devices`, `PATCH /devices/:id`, `DELETE /devices/:id`

**Telemetry (`/telemetry`, role=user):**
- `GET /telemetry` (лог по всіх пристроях користувача з фільтрами:
  `deviceId`, `metricType`, `houseId`, `roomId`, `from`, `to`, `limit`)
- `POST /telemetry` (пакетна вставка для зовнішніх інтеграцій)

**Alerts (`/alerts`, role=user):**
- `GET /alerts` (правила), `POST /alerts`, `PATCH /alerts/:id`, `DELETE /alerts/:id`
- `GET /alerts/active` (активні події), `GET /alerts/events` (історія: active/cleared/all)
- `POST /alerts/events/:id/clear` (ручне зняття алерту)

**Scenarios (`/scenarios`, role=user):**
- `GET /scenarios`, `POST /scenarios`, `PATCH /scenarios/:id`, `DELETE /scenarios/:id`
- `POST /scenarios/:id/run` (ручний запуск)

**Emulator (`/emulator`, role=user):**
- `GET /emulator/status`, `POST /emulator/start` (intervalMs), `POST /emulator/stop`

**Admin (`/admin`, role=admin):**
- `GET /admin/customers` (список з лічильниками будинків/кімнат/пристроїв/сценаріїв/сповіщень)
- `GET /admin/customers/:id` (деталі клієнта)
- `POST /admin/customers/:id/impersonate` (отримати токени клієнта)
- `DELETE /admin/customers/:id` (каскадне видалення)

**Health:** `GET /health` (статус сервера).

### 5.3. Валідація

Уся вхідна валідація — через **Zod**-схеми (наприклад, `name: string().trim().min(1).max(100)`,
`address: max(200).optional().nullable()`). Перевірка прав власності (ownership):
кожен запит фільтрується за `userId` поточного користувача, тому клієнт бачить лише
свої об'єкти.

## 6. Три «рушії» реального часу (бізнес-логіка)

### 6.1. Емулятор телеметрії (`emulator.ts`)

In-process сервіс per-user. Підтримує `Map` таймерів (один на користувача),
інтервал від **1 до 60 секунд** (за замовчуванням 5 с, значення клампиться).

На кожному тіку:
1. Завантажує всі пристрої користувача з останньою телеметрією.
2. Генерує наступне значення **реалістичними генераторами за типом пристрою**:
   - **термостат** (temperature): випадкове блукання ±0.6, клампінг у [18, 26] °C;
   - **лампа** (light_level): базове значення ± до 60, клампінг [100, 600] lx;
   - **лічильник** (power): якщо off → ~0; інакше блукання + 8% шанс стрибка +0.8 kWh;
   - **якість повітря** (co2): блукання +40 з 5% шансом падіння на −200 ppm
     (вентиляція);
   - **датчик руху** (motion): подія `1` з імовірністю 15%, інакше `0`;
   - **розумний замок**: подія з імовірністю 5%;
   - **датчик протікання** (water_leak): подія з імовірністю 1%.
3. Пакетна вставка в `telemetry`.
4. Виклик `evaluateForReadings()` (алерти) та `evaluateSensorScenarios()` (сценарії).
5. Емісія `telemetry:new` по кожному пристрою у WebSocket.

Стійкий до помилок БД (логує і продовжує), очищає стан видалених пристроїв.

### 6.2. Оцінювач алертів (`alertEvaluator.ts`)

`evaluateForReadings(readings)` викликається після кожного запису телеметрії
(емулятор або ручний POST). Алгоритм:
1. Дедуплікація показників до останнього на пару (пристрій, метрика).
2. Для кожного пристрою й активного правила обчислює `value [condition] threshold`.
3. **Rising-edge логіка:**
   - умова виконується + немає відкритої події → створює `AlertEvent`, емітить
     `alert:event` (kind `opened`);
   - умова виконується + подія відкрита → оновлює `latest_value`, `last_seen_at`;
   - умова більше не виконується + подія відкрита → закриває (`cleared_at=now`,
     `clear_reason='auto'`), емітить `cleared`.

### 6.3. Рушій сценаріїв (`scenarioEngine.ts`)

- **`runScenario(id, source)`** — виконує масив дій сценарію:
  - `set_device_status` — змінює статус пристрою, пише в `device_logs`;
  - `notify` — створює запис у `notifications`.
- **Time-тригери** (`startScenarioScheduler`, тік 60 с): для сценаріїв
  `trigger_type='time'` з `{hour, minute}`; дедуп по хвилині (кожен спрацьовує не
  частіше разу на хвилину).
- **Sensor-тригери** (`evaluateSensorScenarios`): rising-edge — спрацьовує, лише
  коли умова стала істинною (попередній стан був хибним).
- Ручний запуск через `POST /scenarios/:id/run`.

## 7. Real-time (WebSocket, Socket.io)

- Ініціалізація на тому ж HTTP-сервері (`initRealtime`).
- **Автентифікація на handshake**: JWT-токен у `socket.handshake.auth.token`,
  перевірка `verifyAccessToken`; при успіху socket приєднується до кімнати
  `user:<id>` (ізоляція подій по користувачу).
- **Вихідні події:**
  - `telemetry:new` — `{ deviceId, points: [{metricType, value, unit, timestamp}] }`
    (live-додавання точок на графіки пристрою й у лог телеметрії);
  - `alert:event` — `opened` / `cleared` (список активних алертів оновлюється без
    перезапиту).

| `/alerts` | Alerts | таби: Активні / Історія / Правила |
| `/settings` | Settings | профіль, аватар, тема, керування емулятором |
| `/admin/customers`, `/admin/customers/:id` | Customers / CustomerDetail | адмін-панель клієнтів |

Захист маршрутів: `ProtectedRoute` (потрібна автентифікація) і `RoleRoute`
(перевірка ролі).

### 8.2. Стан і клієнтська логіка

- **AuthContext** — зберігає JWT у `localStorage`; axios-інтерсептор автоматично
  оновлює access-токен на `401` (single-flight дедуплікація одночасних оновлень).
- **ThemeContext** — тема per-user.
- **api.ts** — типізований axios-клієнт з усіма ендпоінтами й доменними типами.
- **realtime.ts** — обгортка socket.io-client.

### 8.3. Чарти й аналітика (ECharts)

- На сторінці пристрою — live line-chart; для бінарних метрик (motion, water_leak) —
  state-chart на осі 0/1 з лейблами «Спокій / Рух».
- Сторінка `/analytics`: **агрегація** (функції avg/sum/min/max/count + інтервал
  1m..1d; для бінарних форситься `count`; авто-вибір інтервалу під період) +
  4 типи візуалізацій: line, gauge (з кольоровими зонами), bar по днях, heatmap
  (години × дні тижня). Усі графіки перебарвлюються під обрану палітру.

### 8.4. Тема (кастомізація UI)

- **5 brand-палітр** (синя, смарагдова, фіолетова, бурштинова, рожева) — кольорують
  кнопки, посилання, активний пункт меню, графіки.
- **9 primary-палітр** (5 темних: Графіт/Цинк/Камінь/Опівнічний/Темно-зелений;
  4 світлих: Сніговий/Кремовий/Світло-блакитний/Лавандовий) — фон сайдбара й TopBar.
- Збереження per-user: при логіні/імперсонації підтягується з `users.theme_*`,
  при зміні — `PATCH /auth/me/theme`; `localStorage` — кеш для першого рендера.

### 8.5. Адаптивність (mobile)

- Sidebar перетворюється на висувний drawer на ширині `<md` (backdrop + hamburger).
- TopBar згортає email/назву на малих екранах.

## 9. Безпека та авторизація

- **Хешування паролів**: bcrypt (configurable rounds, за замовч. 10).
- **JWT**: access-токен (15 хв) + refresh-токен (7 днів), окремі секрети.
  Access містить `{sub, email, role}`, refresh — `{sub}`.
- **Middleware**: `requireAuth` (перевірка Bearer-токена, `req.user`),
  `requireRole(...roles)` (403 при невідповідності).
- **Розмежування доступу**: доменні ресурси — лише `role=user`; адмін-зона — лише
  `role=admin`. Перевірка ownership по `userId` у кожному запиті.
- **Impersonation (вхід як клієнт)**: `POST /admin/customers/:id/impersonate`
  повертає токени клієнта; фронтенд зберігає токени адміна в резервному слоті
  `localStorage` і показує банер «Повернутись до адміна».
- **Завантаження файлів**: multer з лімітами (аватар до 2 МБ, план до 5 МБ),
  фільтрація типів (PNG/JPEG/WebP); файли віддаються статично за непередбачуваними URL.

## 10. Функціональні можливості (повний перелік фіч)

1. **Автентифікація і ролі** — реєстрація, логін, JWT з auto-refresh, дві ролі.
2. **CRUD-ієрархія** — будинки / кімнати / пристрої з перевіркою власності;
   каскадний вибір (будинок → кімната) у модалках створення.
3. **Плани кімнат** — завантаження зображення плану; розміщення маркерів пристроїв
   на плані drag-and-drop'ом (нормалізовані координати).
4. **Аватар профілю** — завантаження/видалення, відображення в TopBar і Settings.
5. **Телеметрія** — серія для графіка, лог по всіх пристроях, пакетна вставка.
6. **Аналітика** — 4 типи графіків + гнучка агрегація.
7. **Алерти** — порогові правила (рівень будинку), автоматичне відкриття/закриття
   подій (rising-edge), ручне зняття, історія, real-time оновлення.
8. **Сценарії автоматизації** — тригери manual/time/sensor; дії set_device_status
   та notify; час-планувальник + сенсорні rising-edge тригери.
9. **Емулятор** — реалістична генерація телеметрії per-user, керування інтервалом.
10. **Real-time** — WebSocket-оновлення графіків, логів, алертів.
11. **Тема** — 5×9 палітр, persist per-user.
12. **Адмін-панель** — список клієнтів зі статистикою, деталі, видалення, impersonation.
13. **Адаптивний UI** — повна підтримка мобільних екранів.

## 11. Розгортання та запуск

**Передумови:** Node.js 20+, PostgreSQL 16 на `localhost:5432`, БД `diploma`
(користувач/пароль `postgres`).

```bash
# Backend
cd backend
cp .env.example .env
npm install
npx prisma migrate deploy   # застосовує 6 міграцій
npm run dev                 # http://localhost:4000

# Frontend
cd frontend
npm install
npm run dev                 # http://localhost:5173
```

При першому старті автоматично створюються:
- **admin**: `admin@smart-home.local` / `admin12345`
- **demo customer**: `customer@smart-home.local` / `customer12345` — з 1 будинком
  («Квартира на Подолі»), 2 кімнатами (Вітальня, Кухня), 5 пристроями,
  3 правилами алертів і 4 сценаріями.

Vite проксує `/api`, `/uploads`, `/socket.io` → `:4000`.

Корисні команди backend: `npm run build` (tsc), `npm run start`,
`npm run prisma:studio` (GUI БД), `npm run db:reset` (drop+migrate+seed),
`npm run lint`, `npm run format`.

## 12. Можлива структура пояснювальної записки (підказка для генерації)

На основі цього проєкту логічно побудувати такі розділи:

1. **Вступ** — актуальність IoT / розумного дому, мета й завдання роботи.
2. **Аналіз предметної області** — огляд систем розумного дому, аналогів,
   обґрунтування вибору функціональності.
3. **Огляд і вибір технологій** — порівняння стеків (React/Vite, Node/Express,
   PostgreSQL, Prisma, WebSocket), обґрунтування вибору.
4. **Проєктування системи** — архітектура (клієнт-сервер, REST + WebSocket),
   ER-модель БД (11 таблиць), діаграми потоків даних, рольова модель.
5. **Реалізація** — backend (REST API, рушії емулятора/алертів/сценаріїв,
   real-time), frontend (SPA, компоненти, аналітика), автентифікація і безпека.
6. **Тестування та демонстрація** — сценарії використання, демо-дані, скриншоти.
7. **Висновки** — підсумки, можливі напрями розвитку (реальні IoT-пристрої через
   MQTT, мобільний застосунок, ML-аналітика тощо).
